# JARVIS Beta Readiness

## Permanent product boundaries

MT5 is not connected. Broker execution and automatic trading are not supported. These are permanent product boundaries for Beta and V1, not future sprint items. JARVIS uses one Core Agent and read-only deterministic tools.

## Production commands

```powershell
npm ci
npm run build
npm start
```

`npm start` loads `.env` when present and starts the local production-compatible server. Set `HOST=0.0.0.0` only when a trusted reverse proxy or container network requires it. Verify startup with:

```powershell
Invoke-RestMethod http://127.0.0.1:4174/health
Invoke-RestMethod http://127.0.0.1:4174/api/status
npm run smoke:production
```

OpenAI Sites deployment continues to use `.openai/hosting.json` and the worker generated by `npm run build`. Deployment is intentionally manual.

## Closed Beta access

Set `BETA_MODE=true`, `BETA_ALLOWED_EMAILS`, `BETA_INVITE_CODES`, and `BETA_MAX_USERS`. The first Beta uses server-side HTTP Basic authentication: the username is an allowlisted email and the password is an invite code. Browser credentials are sent only to the same HTTPS origin. The visible Demo login password is not an authentication control and cannot bypass the server gate.

When Beta mode is enabled with an empty email or invite-code list, readiness fails closed. Do not put real values in `.env.example`, source code, browser bundles, logs, or documentation.

## Environment classes

- Base: `NODE_ENV`, `HOST`, `PORT`, `APP_BASE_URL`, `BUILD_VERSION`.
- Beta: `BETA_MODE`, `BETA_ALLOWED_EMAILS`, `BETA_INVITE_CODES`, `BETA_MAX_USERS`.
- Market Data: `MARKET_DATA_MODE`, `MARKET_DATA_PROVIDER`, `TWELVE_DATA_API_KEY`, `TWELVE_DATA_BASE_URL`.
- OpenAI text: `AI_MODE`, `AI_PROVIDER`, `OPENAI_API_KEY`, `OPENAI_MODEL`, `OPENAI_BASE_URL` and bounded tool/token settings.
- OpenAI Vision: `VISION_MODE`, `VISION_PROVIDER`, `OPENAI_VISION_MODEL` and upload limits.
- Macro and News are optional provider foundations and remain Demo or unavailable without live adapters.

Missing optional providers do not stop the base application. They must remain visibly Demo, partial, or unavailable and must never appear Live.

## Security and operations

The platform wrapper adds request IDs, safe response headers, exact-origin CORS, body-size checks, method restrictions, generic production errors, server-side per-user limits, and a bounded in-memory audit trail. Logs are structured JSON on stdout/stderr. They exclude credentials, invite codes, prompts, conversations, image bytes, local paths, and hidden reasoning.

Health endpoints are `GET /health` and `GET /api/status`. Status checks do not call OpenAI or retry Vision. OpenAI Vision remains implemented but live verification is blocked by provider quota.

## Known limitations

- Beta authentication uses HTTP Basic and requires HTTPS. It is deliberately smaller than a full identity platform.
- Beta user capacity, rate-limit buckets, audit entries, feedback, conversations, scanner results, uploads, and analyses are memory-local and reset on process restart.
- Vision uploads remain owner-isolated and temporary, but restart removes them. Users must upload again after restart.
- Local memory limits are not distributed protection; use one application instance for the first Beta.
- Macro and News live providers are not connected.
- OpenAI Vision live verification is blocked by `insufficient_quota`; hardening must not retry it without explicit approval.
- Vite reports that the classic `app.bundle.js` cannot be bundled without `type="module"`. The approved runtime remains functional through the explicit static worker packaging. Changing execution mode could alter script ordering, so the warning is retained rather than risking the locked UI.

## Rollback

Stop the process, restore the previous Git commit, run `npm ci && npm run build`, then start again. Do not copy `.env` into source control. Runtime logs are emitted to the hosting platform or process supervisor; local Codex/Vite logs remain ignored.
